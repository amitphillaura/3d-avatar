Exposing a WSL2 Service Over Tailscale
(Without Exposing It to Anything Else)

A service running inside WSL2 on a Windows laptop, reachable from your Mac over Tailscale — and from nothing else. Mirrored networking, binding the Tailscale IP, a scoped firewall rule, and the silent idle-shutdown trap that kills always-on WSL servers.

TL;DR

You have a service (an HTTP API, say on port 5190) running inside WSL2 on a Windows machine, and you want to reach it from another device over Tailscale — but only over Tailscale. Four things have to be true:

  1. WSL must be able to bind the Tailscale IP. Default NAT networking can't — switch to networkingMode=mirrored.
  2. Bind the service to the Tailscale IP specifically (not 0.0.0.0) so Wi-Fi/LAN can't reach it.
  3. Open the firewall for that port — scoped to the Tailscale range (100.64.0.0/10), nothing wider.
  4. Keep the WSL2 VM alive. It idle-shuts-down by default, and an incoming request cannot wake a powered-off VM. This is the one that silently bites you.

Why WSL2 makes this hard

Tailscale runs on the Windows host. Your service runs in the WSL2 guest, which by default sits behind a NAT with its own private network (172.x). WSL forwards localhost between Windows and the guest, so 127.0.0.1:5190 works from Windows — but that's it. A packet arriving from the Mac on the host's Tailscale IP has no route into the NAT'd guest. The service is invisible to the tailnet.

Fix 1 — mirrored networking + bind the Tailscale IP

WSL2's mirrored networking mode shares the Windows host's network interfaces directly into the guest. The Tailscale adapter's IP then appears inside WSL, so your service can bind it. Add to %USERPROFILE%\.wslconfig:

[wsl2]
networkingMode=mirrored

[experimental]
hostAddressLoopback=true   # let the Windows host reach WSL via the mirrored IPs

Then wsl --shutdown (it reads .wslconfig on next start). After it restarts, the Tailscale IP is a real interface inside WSL:

# inside WSL
$ ip -4 addr show | grep inet
    inet 100.83.31.91/32 ... eth0      # <- the Tailscale IP, now bindable
    inet 192.168.4.105/22 ... eth1     # the Wi-Fi/LAN IP

Now bind the service to the Tailscale IP specifically — not 0.0.0.0. Binding the exact IP means the LAN/Wi-Fi address simply isn't served; only the tailnet interface answers. With systemd, a machine-local drop-in keeps this out of your committed unit:

# /etc/systemd/system/<your>.service.d/override.conf
[Unit]
StartLimitIntervalSec=0          # retry forever (Tailscale IP may be late at boot)

[Service]
Environment=HOST=100.83.31.91    # your service's bind-host var
Restart=on-failure
RestartSec=5

(StartLimitIntervalSec=0 matters: at boot the service may start before Tailscale has assigned the IP, so the first bind fails — let systemd keep retrying instead of giving up after the default 5 tries.)

Fix 2 — a firewall rule, scoped to the tailnet

Windows Firewall default-blocks unsolicited inbound, and in mirrored mode it governs WSL traffic too. So the Mac's connection is dropped until you allow the port — but allow it only from the Tailscale CGNAT range, never wider. In an elevated PowerShell:

New-NetFirewallRule -DisplayName 'My API (Tailscale 5190)' `
  -Direction Inbound -Action Allow -Protocol TCP -LocalPort 5190 `
  -RemoteAddress 100.64.0.0/10 -Profile Any

100.64.0.0/10 is Tailscale's address range. Combined with binding the Tailscale IP, that's defense in depth: the socket only listens on the tailnet interface, and the firewall only admits tailnet sources.

The trap — WSL2 idle-shutdown

This is the one that wastes an afternoon. After everything above, the service worked… then didn't… then did again. The journal told the story:

Started my-api.service
node[539]: Server listening at http://100.83.31.91:5190
node[539]: incoming request /health -> 200      # it works!
Stopping my-api.service...
Deactivated successfully.                        # ...graceful stop, no crash
Started my-api.service                            # ...12s later, back again

Not a crash — a graceful stop. WSL2 powers the whole VM off when it's idle (no active wsl.exe session), which cleanly shuts systemd down with it. The killer detail for a server: an incoming network request does not wake a powered-off WSL VM. Only a wsl command or Windows-localhost access starts it. So your "always-on" API is actually off whenever nobody's looking — and the Mac just sees a dead port.

The fix: stop it idling, and hold a session open

Two layers. First, tell WSL not to idle-shut-down, in .wslconfig:

[wsl2]
networkingMode=mirrored
vmIdleTimeout=-1            # never power off the VM for being idle

Second — and this is the reliable part — hold a WSL session open so there's always a live process keeping the VM warm. A hidden launcher in the Windows Startup folder (no admin needed) starts the service and then parks on sleep infinity:

' %APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup\KeepAlive.vbs
Set sh = CreateObject("WScript.Shell")
sh.Run "wsl.exe -d Ubuntu -u root -- /bin/sh -c ""systemctl start my-api; exec sleep infinity""", 0, False

The 0 runs it with no window; sleep infinity never exits, so the VM never goes idle. It fires at every logon and is a no-op if the service is already up.

Caveat — "always-on" means "while logged in." This keeps WSL alive for the logged-in user. To survive logout/the login screen you'd need to run WSL from a true Windows Service (e.g. via NSSM or Task Scheduler "run whether logged on or not") — more moving parts. For a personal box you sign into, the Startup launcher is enough.

The security model

Three independent layers keep this tailnet-only. Any one would help; together they fail safe:

Bind the Tailscale IP
socket listens on 100.x only — LAN/Wi-Fi address isn't served
Firewall 100.64.0.0/10
inbound port admitted only from tailnet peers
Tailscale ACLs
restrict which tailnet devices may reach the port
No port-forward
router never forwards the port; nothing is public

Note what we did not do: bind 0.0.0.0. That would expose the port on every interface — including whatever coffee-shop Wi-Fi the laptop joins — and lean entirely on the firewall being correct. Binding the specific Tailscale IP is the safer default for a machine that roams.

The full recipe

  1. %USERPROFILE%\.wslconfig 
    [wsl2]
networkingMode=mirrored
vmIdleTimeout=-1

[experimental]
hostAddressLoopback=true
  2. Restart WSL: wsl --shutdown
  3. Get the Tailscale IP: tailscale ip -4 (e.g. 100.83.31.91)
  4. Bind it via a systemd drop-in at /etc/systemd/system/<svc>.service.d/override.conf (the [Unit]/[Service] block above).
  5. Firewall (elevated PowerShell): the New-NetFirewallRule … -RemoteAddress 100.64.0.0/10 above.
  6. Keep-alive launcher in the Startup folder (the KeepAlive.vbs above).
  7. Start it: run the VBS (or systemctl start <svc> + a held-open session).

Verify

# inside WSL — bound to the Tailscale IP, not 0.0.0.0?
$ ss -tlnp | grep 5190
LISTEN 0 511 100.83.31.91:5190 0.0.0.0:* users:(("node",pid=126))

# from the Windows host (needs hostAddressLoopback)
> curl http://100.83.31.91:5190/health      # {"ok":true,...}

# from the Mac (the real test) — by IP or MagicDNS name
$ curl http://100.83.31.91:5190/health
$ curl http://amitlaptop:5190/health        # {"ok":true,...}

# confirm the LAN IP is NOT served (should refuse)
> curl http://192.168.4.105:5190/health     # connection refused — good
