TL;DR

  • Goal: single photo → static textured .glb, served from a home RTX 4070 laptop over a submit/poll/download HTTP API, auto-starting on boot, local-only.
  • Engine: TripoSR as a smoke test — engine-agnostic design, so SF3D / Hunyuan3D drop in behind the same flag later.
  • Runtime: the whole backend (Node + Python) runs in WSL2 Ubuntu; GPU via WSL CUDA passthrough.
  • The hard part was the dependencies: torchmcubes needs nvcc (we don't have it) → CMake find_package(Torch) hard-requires CUDA → replaced with a compile-free PyMCubes CPU shim; plus a transformers major-version bump that broke checkpoint loading.
  • Result: photo → .glb (42,076 verts, watertight, ~54 s on cuda:0), end to end through the API, as a systemd service that survives reboots.

The goal & design decisions

We wanted a small service that turns a single photo into a 3D model, running on a home machine and callable from our app over Tailscale. It slots into an existing repo whose backend is Fastify + better-sqlite3 + Python workers. A few choices shaped everything:

  1. Engine-agnostic from day one. The API dispatches on an engine flag; each model lives behind an isolated adapter. TripoSR is the smoke-test engine, not the destination.
  2. A job queue, not a synchronous request. Generation takes tens of seconds. The contract is submit → poll → download.
  3. Mirror the existing backend exactly — a new route module, a mesh_jobs table, and a Python worker spawned via spawn(PYTHON_BIN, …), down to the stdout protocol (PROGRESS:<n> lines + a final JSON status line).
  4. WSL2, with the whole backend inside it. Node runs in WSL too — otherwise spawn('.venv/bin/python3') wouldn't resolve.
  5. Local only. Bind 127.0.0.1, enforce a 25 MB image limit, validate content types. Tailscale later; nothing exposed publicly.

The data model is one table:

CREATE TABLE mesh_jobs (
  id TEXT PRIMARY KEY,
  status TEXT NOT NULL DEFAULT 'queued',   -- queued|running|ready|failed
  stage  TEXT NOT NULL DEFAULT 'mesh',     -- mesh|rig
  engine TEXT NOT NULL,                     -- triposr|sf3d|hunyuan3d
  source_path TEXT NOT NULL,
  result_path TEXT, vrm_path TEXT,
  error TEXT, params_json TEXT,
  created_at TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);

Architecture

The build

The Node side went in cleanly because it copied conventions that already existed: backend/lib/paths.js (mesh path helpers), backend/lib/mesh.js (job lifecycle + worker spawn + stale-job recovery), backend/routes/mesh.js (the /api/mesh/* endpoints), and backend/worker/generate_mesh.py (engine dispatch — each adapter imports its heavy deps inside the function so a missing engine never breaks the others).

We validated the whole Node layer before touching Python. On the Windows host (Node 22 already present), booting the server and curling the endpoints proved the schema, routes, validation (201 / 409 / 415), the job lifecycle, and DELETE — without a single GB downloaded. That's the cheap confidence you want before committing to multi-GB installs.

The dependency odyssey

Here's where a "few-days" task earns its war stories. Every one of these was a real failure with a fix.

0 · Ubuntu 26.04 ships Python 3.14 — too new for ML

No PyTorch wheels for 3.14. We used uv to manage a Python 3.11 toolchain and venv — no system Python touched. torch installed from the cu121 wheels (newer than the box's CUDA 13.1 driver, which is fine — drivers are backward compatible). GPU passthrough worked out of the box: torch.cuda.is_available() → True.

1 · torchmcubes wants nvcc (and an atomic installer hides it)

TripoSR's only compiled dependency is torchmcubes (marching cubes). Its build checks torch.cuda.is_available() — True in WSL — and tries to compile a CUDA extension. But WSL has no nvcc. The build failed, and because uv pip install -r is atomic, that single failure rolled back the entire requirements file (even pure-Python packages were absent — a confusing symptom).

2 · …but find_package(Torch) drags in a hard CUDA requirement

Hiding the GPU (CUDA_VISIBLE_DEVICES="") got CMake to pick the CPU path, but the newer torchmcubes uses CMake/scikit-build-core and does find_package(Torch). Build isolation hid the venv's torch (couldn't find TorchConfig.cmake); fixing that with --no-build-isolation + CMAKE_PREFIX_PATH then hit Caffe2's hard CUDA requirement — because our torch is the cu121 build, the config insists on CUDA libraries at configure time, even for a CPU-only extension. No CUDA toolkit → dead end.

3 · The fix: a compile-free CPU shim

TripoSR only needs marching_cubes(level, 0.0) → (verts, faces). So we replaced torchmcubes with a tiny drop-in shim backed by PyMCubes (manylinux wheel — no compiler). It installs into the venv as torchmcubes, so TripoSR's import resolves to it and the engine source stays untouched:

# backend/engines/torchmcubes_shim.py
import mcubes, numpy as np, torch

def marching_cubes(vol, isovalue=0.0):
    arr = vol.detach().cpu().numpy().astype(np.float64)
    verts, faces = mcubes.marching_cubes(arr, float(isovalue))
    verts = verts[:, [2, 1, 0]]          # PyMCubes (i,j,k) -> torchmcubes (k,j,i)
    return (torch.from_numpy(np.ascontiguousarray(verts)).float(),
            torch.from_numpy(np.ascontiguousarray(faces)).long())

The [2, 1, 0] swap matters: PyMCubes returns vertices in array-index order (i,j,k); the real torchmcubes returns (k,j,i); and TripoSR applies its own [2,1,0] downstream. Match the convention here and the geometry comes out correct (a watertight mesh), not mirrored.

4 · The transformers trap (loose pins bite at load time)

Our requirements had loose pins (transformers>=4.35). Installed after TripoSR's pinned 4.35.0, they silently upgraded it to 5.12.1. Imports still worked — so it looked healthy. Then the weights failed to load:

Missing key(s):    image_tokenizer.model.layers.0.attention.q_proj.weight …
Unexpected key(s): image_tokenizer.model.encoder.layer.0.attention.attention.query.weight …

transformers 5.x refactored ViTModel's state-dict layout, so TripoSR's checkpoint (saved against 4.x naming) couldn't load. Fix: pin back to 4.35.0 (with matching tokenizers==0.14.1, huggingface-hub==0.17.3).

Lesson: never re-pin packages an engine already pins. We rewrote requirements-mesh.txt to own only what the engine doesn't (onnxruntime, PyMCubes) and to explicitly not list transformers etc.

5 · One more: rembg made onnxruntime optional

Background removal failed with No module named 'onnxruntime'. Recent rembg moved it to an optional extra, so we add onnxruntime explicitly (CPU is plenty fast for single-image masking).

Making it a service that starts on boot

WSL2 only boots on demand, so an enabled service won't run until something starts the distro. The clean answer is two small pieces:

  1. A systemd unit inside WSL (mesh-api.service) with Restart=on-failure, binding 127.0.0.1:5190, installed and enabled. This is the real service.
  2. A Windows logon launcher — a hidden VBS in the user's Startup folder that runs wsl -d Ubuntu -u root -- systemctl start mesh-api at sign-in. It boots WSL, systemd brings up the enabled service, and it's a no-op if already running. No admin required (registering a Scheduled Task needs elevation; the Startup folder doesn't).

Ordering subtlety: mesh.js resolves PYTHON_BIN (.venv/bin/python3 vs fallback) once at startup. Start the service before the venv exists and it caches the wrong interpreter. So: finish the install, then start the service.

What Phase 1 buys, and what's next

Phase 1 is shippable on its own: any photo → a static textured .glb you can load in a Three.js GLTFLoader, served from a home GPU over a job-queue API that starts on boot and stays local until we bind it to Tailscale.

The honest caveat for the end goal: every image-to-3D engine emits a static, unrigged mesh in the photo's pose. None produce a VRM humanoid. The mesh step is ~30% of the work; rig → VRM → retarget is the other ~70% — that's Phase 2, where the real risk lives.

v2 — a self-describing API

First contact with a real client (the app calling this over Tailscale) produced the most useful feedback of the project: the API worked, but it made the client do too much guessing. So v2 makes the job record carry everything a client needs — no downloading-and-parsing the GLB to find out what you got.

  • Numeric progress (0–100) on running jobs — a real bar, not a spinner.
  • Result metadata inline: bytes, vertices, triangles, bbox, dimensions, vertex_colors, sha256.
  • Honest params: params_requested vs params_applied. Asking for texture:true on TripoSR reports back {texture:false, color_source:"vertex"} — because it bakes vertex colors, not a UV material. Don't let a flag lie.
  • Timing: started_at, finished_at, duration_ms, queue_wait_ms.
  • Content-Length + ETag (the GLB's sha256) with If-None-Match → 304, so clients get download progress and free caching.
  • Structured errors: error:{code, message, stage} (e.g. bad_image, oom, model_load_failed) instead of a bare string.
  • No path leaks: result_url / thumbnail_url / input_url replace internal filesystem paths.
  • Idempotency: a sha256 of image+params dedups identical requests — resubmitting returns the existing job instead of burning 60 s of GPU.
  • One-GPU queue with position + ETA when queued; /health reports busy, queue_depth, used VRAM, and capability flags (phase, rig_available, max_image_bytes) so clients discover features rather than assume them.
  • Server-rendered thumbnail + the background-removed input the engine actually saw — job-history previews without booting a 3D viewer.

The orientation trap (claiming Y-up ≠ being Y-up)

One v2 feature — "normalize the mesh: centered, unit-scaled, Y-up" — shipped with a bug worth calling out, because it's the classic axis pitfall. The first pass centered and scaled the mesh and reported up_axis:"Y" — but never actually reoriented it. The tell was in the data it returned: dimensions showed the tall axis was Z while it claimed Y. TripoSR emits Z-up; glTF/Three.js are Y-up, so the model rendered lying down.

The fix wasn't to guess a rotation — it was to render the four cardinal sides and look. Upright came from Rx(-90) (Z→Y); the chair's front turned out to face +X, so a second Ry(-90) brought it to +Z. Canonical transform: Ry(-90)·Rx(-90) → height +Y, front +Z, and now up_axis/front_axis are true. Lesson: a metadata field that asserts an invariant is a liability until something enforces it — and for orientation, "render it and look" beats reasoning every time.
